IMPORTANT: TENANT vs SCHOOL (FUTURE-PROOFING)
==================================================
To support future scenarios like:
- One owner managing multiple schools
- Comparing two schools (benchmarking)
- Merging/acquiring schools
Define terminology explicitly:
TENANT
- The billing and isolation boundary for a customer account.
- A tenant can represent:
  A) One school (most common for SMB pricing), OR
  B) A school group/organization that contains multiple schools/campuses (enterprise)
SCHOOL / CAMPUS (INSIDE TENANT)
- A functional unit inside a tenant (campus/branch/school unit).
- Recommended to model as a first-class entity so:
  - One tenant can contain multiple schools
  - Cross-school comparison is possible within the same tenant safely
CROSS-SCHOOL (CROSS-TENANT) ACCESS
- Regular school users must NEVER access other tenants’ data.
- Any cross-tenant comparison must be:
  - Super Admin only, or
  - Explicitly granted via a governance-controlled relationship model
  - Audited and revocable
SCHOOL MERGE APPROACH (PREFERRED)
- Prefer “Group multiple schools under one tenant” over merging tenants.
- If tenant merge is required, it must be treated as an admin-only data migration workflow with:
  - Dry-run preview
  - Conflict resolution
  - Full audit logging
  - Background job execution + rollback checkpoints

MULTI-TENANCY ARCHITECTURE (MUST DECIDE & DOCUMENT)
==================================================
Choose ONE primary tenancy strategy and implement consistently across the platform:
OPTION A (RECOMMENDED EARLY) — SINGLE DB + SHARED SCHEMA (ROW-LEVEL TENANCY)
- Every tenant-owned table contains tenant_id
- Tenant isolation enforced at:
  - Request tenant resolution (domain/subdomain/header/token claim)
  - Service/repository layer tenant filters
  - Authorization checks + permission evaluation
- Tenant-aware indexing approach:
  - Composite indexes starting with tenant_id for high-cardinality queries
- Automated tenant isolation test suite:
  - Cross-tenant read/write attempts must be blocked deterministically
OPTION B — SCHEMA PER TENANT
- Requires schema orchestration for migrations, backups, and reporting
OPTION C — DATABASE PER TENANT
- Provides the strongest isolation with automated provisioning, upgrades, and monitoring
RECOMMENDATION
- Default for CloudCampus: OPTION A (single DB + shared schema + tenant_id) for most schools to keep onboarding and operations cost-effective at 1000+ tenants.
- Offer OPTION C only as an Enterprise / Dedicated tier when a tenant pays for dedicated isolation, custom backup/restore expectations, or special compliance.
- Use OPTION B only if you want stronger isolation than OPTION A but still want to avoid the per-tenant infrastructure overhead of OPTION C.
Minimum required documentation regardless of option:
- Tenant resolution precedence rules (custom domain > subdomain > header > token claim, or defined order)
- Data migration strategy (how schema changes roll out safely)
- Backup/restore strategy (including tenant-level restore expectations)

AI TECHNOLOGY LAYER (ENTERPRISE AI PLATFORM)
==================================================
AI GOAL
Deliver safe, configurable, tenant-scoped AI capabilities across:
- Super Admin governance
- School Admin ERP
- Teacher / Parent / Student experiences
- Website Builder & Digital Experience Platform
AI MUST BE
- Feature-flagged and subscription-controlled per tenant
- Fully configuration-driven (no tenant-specific code forks)
- Tenant-scoped for data, prompts, and knowledge bases
- Audited (who asked what, which data sources were used, which actions were taken)
- Cost-controlled (per-tenant budgets, rate limits, and usage metering)
AI PLATFORM BUILDING BLOCKS
1) AI GATEWAY SERVICE (MODEL PROVIDER ABSTRACTION)
- One internal service for all AI calls (chat, embeddings, reranking, and future multimodal)
- Provider-agnostic routing (support multiple LLM providers behind one interface)
- Per-tenant routing policies (latency/cost tiers, allowed models)
- Per-tenant usage tracking (tokens, requests, latency)
2) PROMPT & POLICY REGISTRY (VERSIONED + UI CONTROLLED)
- Prompts stored as versioned configuration (draft → publish → rollback)
- Role-aware instructions (Super Admin vs School Admin vs Teacher vs Parent vs Student)
- Structured outputs for automation flows (JSON schemas for actions and data extraction)
- Rollout controls (beta/canary by tenant, with feature flags)
3) TENANT KNOWLEDGE BASE (RAG)
- Default approach: retrieval-augmented generation (RAG) for tenant documents and platform knowledge
- Pipeline:
  - Upload/source connect → parse → chunk → embed → index
- Metadata tagging on every chunk:
  - tenant_id, school_id, module, audience, permissions, source_type, updated_at
- Retrieval rules:
  - Always filter by tenant_id (and school_id where applicable)
  - Support permission-aware retrieval (role-based visibility at chunk level)
- Response rules:
  - Provide citations to source documents/sections wherever possible
4) AI TOOLS / ACTIONS (ERP + WEBSITE AUTOMATION)
- AI can call internal APIs through a controlled tool layer:
  - Create leads, draft notifications, generate reports, summarize logs, assist admins
- Actions must be:
  - Permission-checked (RBAC)
  - Logged (audit trail)
  - Optionally approval-gated for sensitive operations (workflow engine integration)
5) AI QUALITY + CONTINUOUS IMPROVEMENT
- Maintain an evaluation dataset per module (questions + expected behavior)
- Track quality signals:
  - user feedback (thumbs up/down), citation coverage, response latency, cost per request
- Prompt and retrieval configuration changes should be tested before wide rollout
6) AI COST & SAFETY CONTROLS (TENANT LEVEL)
- Meter AI usage per tenant (tokens, tool calls, storage for embeddings)
- Enforce plan-based limits (daily/monthly budgets, rate limits, concurrency)
- Provide Super Admin visibility:
  - top tenants by usage, cost, and adoption, with ability to pause AI per tenant
AI IMPLEMENTATION SEQUENCE (RECOMMENDED)
1. Build AI gateway service + tenant-scoped routing
2. Add prompt/policy registry with versioning + rollout control
3. Implement tenant knowledge base (RAG) with permission-aware retrieval
4. Release internal copilots first (Super Admin + School Admin) for controlled adoption
5. Expand to Teacher/Parent/Student assistants + Website AI modules
6. Add AI analytics insights + automation workflows (approval-gated where needed)
